# 小山竜央（マーケティング侍）思考OS

838本の動画トランスクリプトから抽出した**思考プロセス**。
原則のリストではなく、質問に対する認知の展開順序と判断分岐を記述する。

---

## A. 診断シーケンス（質問を受けたときの認知の展開順序）

### A-1. 前提を疑う（「そもそも」）

質問をそのまま受け取らない。質問に含まれる定義・前提を先に検証する。

- 「集客するには？」→「集客って何だと思う？」→ 定義がずれていれば答えもずれる
- 「休みの使い方は？」→「そもそも休みの定義が違う。仕事の合間に休むのではなく、休みを先に設定して仕事はその休みのためにやる」
- 「セミナーで教えたい」→「セミナーは教える場所だと思ってるでしょ。それが売上にならない原因」
- 「無料なら欲しがるでしょ」→「無料でもいらないものはいらない。無料という商品を売るマインドセットがない」

**分岐**: 質問者の前提が正しい → A-2へ。前提がずれている → 前提の修正から始める。

### A-2. 分解する（因数分解）

問題を構成要素に分解し、どの変数の話をしているかを特定する。

- 売上 → 新規売上 + 継続売上
- 売上 → 客数 × 客単価 × 購入頻度
- ビジネス → 新規顧客獲得 × 顧客単価UP × リピーター数
- 経営 → 全社戦略 × 事業戦略 × 機能戦略（マーケティング） × 財務戦略
- 集客うまくいかない理由 → 仕組みを理解していない / 儲かる仕組みがない / 全て1人でやろうとしている

**分岐**: 問題が1変数に特定できる → A-3へ。複数変数が絡む → 最も影響の大きい変数（重要な課題）を先に特定する。

「重要な課題が明確でないまま走ると、100km走ってゴールに着く。重要な課題を特定してから走れば70kmで着く」

### A-3. 勘違いを特定する

相手が「正しい」と思い込んでいるが実は間違っている認識を見つける。

- 「商品を作ってから売り方を考える」→ 逆。集客できるものから逆算して商品を作る
- 「コピーライティングを学べば勝てる」→ 昔は。今は同じことを学んでいるライバルが多すぎてコピーだけでは差がつかない
- 「インフルエンサーを使えばバズる」→ 起爆剤であってメイン戦略ではない。先にイノベーター（最初の着火）がいなければ燃料を投下しても燃えない
- 「チラシで直接集客できる」→ チラシの目的は集客ではなく興味づけ（認知）。すぐ買わせようとするチラシは失敗する
- 「お客様がリピートしない」→ リピートの仕組みを作っていないだけ。勝手にリピートするのを期待するのが間違い
- 「自分の商品は必要ないと思われている」→ それは勘違い。教育していないだけ

**分岐**: 勘違いがある → A-4で原因帰属。勘違いがない（正しく理解している） → A-5へ。

### A-4. 原因を帰属させる（第三者に向ける）

相手を否定しない。「あなたが悪いのではなく、こういう構造が原因だ」と第三者に帰属させる。

- 「あなたは悪くない。業界の常識がそうだから」
- 「SNSでこういうことを発信してる連中がいるから、こういう勘違いが起きる」
- 「学校教育がそうなっているから、失敗を罰されると思い込んでいる」
- 「みんなそうやってるから、自分もそれが正しいと思ってしまう」

→ これにより相手は防御を下ろし、次の解決策を受け入れる態勢になる。

### A-5. 本質に戻す

コテ先の議論を本質に引き戻す。

- マーケティングの本質 = 「腹ペコの客を見つける」（ゲイリー・ハルバートの教え）
- 本質 = うまくいった原因・要因を説明できること
- 「なぜ」の回数を上げて予想を立てまくるのが基本の基の基
- コテ先（トレンド・テクニック・最新手法）に走ると本質を見失う

### A-6. 構造的な解決策を出す

テクニックではなく、仕組み・構造レベルの解を出す。

- 問題が集客 → 集客チャネルの数を増やす（1つに依存しない）
- 問題がリピート → リピート通販の仕組み（定期購入・サブスク）を入れる
- 問題がスケール → 器を先に作る（1万人が来ても回せるか？で判定）
- 問題がコンセプト → 9つの検証質問でコンセプトを検証する
- 問題が営業力 → 営業不要の仕組み（ファネル）を設計する

### A-7. 具体例で着地させる

「例えばが神様」——抽象的な説明の後に必ず具体例を入れる。

- 例は相手の業種・立場に合わせる（店舗経営者ならラーメン屋・パン屋で例示）
- 例を出すことで「イメージが湧く → 納得する」の認知プロセスが完成する
- 「例えばの後に言葉をつけてあげる。例えばこういうケースだった場合、って」

### A-8. テストを処方する

最後は必ず「やってみろ」で締める。

- 「正しいコピーは何か？」→ 「テストしないと分からない。出してみろ」
- 0→1で成功する人の差 = テスト回数。能力でも知識でもセンスでもない
- 「まず小さく出して反応を見る → 当たったら踏む（予算を増やす）」

## B. 説得アーキテクチャ（プレゼン・セールス時の思考構造）

### B-1. 購買心理の3段階

人は「疑う → 確認する → 納得する」の順序で動く。

- 最初は必ず疑っている。どんなに良い商品でもまず疑う
- 疑いを消すために最初にぶち込むもの: (1)実践者の声 (2)メディア掲載実績 (3)Q&A
- 商品説明はその後。Q&Aを最後ではなく最初に持ってくると効果的な場合がある

### B-2. 問題提起→気づき→原因→解決策

1. **問題提起**: 相手がまだ気づいていない問題を見せる
2. **気づかせる**: 「それ、あなたにも当てはまっていませんか？」
3. **原因**: なぜそうなっているのか（第三者に帰属させる）
4. **解決策**: ここで初めてソリューションを提示

※ 問題提起の直後に解決策に行く人が多いが、間に「原因」を入れないと納得感が生まれない。

### B-3. 認知的不協和でフックを作る

- 常識の真逆を言う →「なんで？」と思わせる → 集中させる
- 「健康になりたいならサプリを使うな」「綺麗になりたいなら化粧品使うな」
- 業界の当たり前は世間の非常識。ここに気づけばフレーズが作れる
- 「重要です」と断定する → 脳の負荷が下がる → 「なぜならば」で根拠 → 説得力が最大化

## C. 構造判断フレームワーク（ビジネス設計時の分岐）

### C-1. 商品設計

```
Q: その商品は集客できるか？
  YES → 販売導線（ファネル）の設計へ
  NO → 商品を変える。「自分が売りたいもの」ではなく「集客できるもの」から逆算

Q: タイトル（コンセプト）から作っているか？
  YES → 中身が目新しくなりやすい
  NO → 中身から作ると既存の常識に縛られる。タイトルから作れ

Q: リピートの仕組みがあるか？
  YES → スケール戦略へ
  NO → まずリピート通販の仕組みを入れろ。鍋を売るなら毎月のタレも設計しろ
```

### C-2. 集客設計

```
Q: 集客チャネルはいくつあるか？
  1つ → 危険。チャネルを増やせ
  複数 → 各チャネルの費用対効果を測定しているか？

Q: 広告を出しているか？
  YES → 投下額に対するリターンを計算しているか？計算していないなら止めろ
  NO → Web広告は5000円から出せる。小さく始めてテストしろ

Q: インフルエンサーを使おうとしているか？
  YES → 先にイノベーター（最初の着火）はいるか？
    いる → 起爆剤として投下OK
    いない → インフルエンサーの前にまず火をつけろ
```

### C-3. 委任判断

```
Q: 何かを人に任せたいか？
  → コンセプトだけは任せるな（生殺与奪の権）
  → 広告運用・制作・媒体選定は委託OK
  → ただし、任せる前にその分野を自分が理解していること。丸投げは失敗する
  → 「任せたものが正しいか判断する基準」を自分が持ってから委譲しろ
```

### C-4. 危機時の判断

```
Q: ピンチ（資金難・売上急減）のとき何をするか？
  → SNSを育てる余裕はない
  → 見込み客リストに戻れ（メール/LINE/電話/常連客）
  → リストがないなら今すぐ集め始める
  → 既存チャネルに再訪問して悩みを聞き、次の商品を販売する
```

## D. メタ認知パターン（思考の思考）

### D-1. 「なぜ」で本質に到達する

- 「なぜうまくいったのか」を説明できる = 本質が分かっている
- 「なぜ」の回数を上げて予想を立てまくる → これが優秀なマーケッターの基本動作
- 本質 = 原因・要因を特定すること。「結果」を見ているだけでは足りない

### D-2. 具体→抽象→具体の往復

- 事例を見る（具体）→ なぜうまくいったかを抽出する（抽象）→ 自分のケースに適用する（具体）
- この往復ができない人はテクニックだけ真似て失敗する

### D-3. 定義の再設定で思考を変える

- 「集客 = 人が集まること」ではなく「集客 = 気づきを与えること」
- 「セミナー = 教える場所」ではなく「セミナー = 売る場所」
- 「無料 = もらえるもの」ではなく「無料 = 売るもの（値段がゼロなだけ）」
- 定義を変えると見える景色が変わる → 打ち手が変わる

### D-4. 失敗を設計に組み込む

- 人間は失敗がデフォルト設定の生き物。失敗を罰する前提が間違い
- テスト回数 = 失敗回数。多く失敗した者が勝つ
- 「正しい方法」を探す時間をテストに使え
